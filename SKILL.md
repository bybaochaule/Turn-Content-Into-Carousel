# Turn Content Into Carousel

## Core Standard

Turn source content into a clear, swipe-worthy carousel storyboard that can be handed to a designer, pasted into a document, or built manually later. Focus on message architecture, slide copy, pacing, visual cues, captions, and quality control, not graphic design production.

Default to a complete carousel package unless the user asks for only hooks, an outline, an audit, or a shorter draft.

## Workflow

1. Confirm the input and destination:
   - Source: article, transcript, notes, thread, report, outline, URL text, or rough idea.
   - Platform: LinkedIn, Instagram, TikTok photo-mode, PDF, presentation, or generic carousel.
   - Audience, voice, CTA, slide count, and any brand or formatting constraints.
   - If missing details are not blocking, assume LinkedIn, 8 slides, practical-professional tone, and a soft CTA.

2. Extract the content backbone:
   - Identify the main promise, reader problem, strongest insight, proof points, examples, steps, objections, and final takeaway.
   - Preserve the source's meaning. Do not invent facts, statistics, quotes, outcomes, screenshots, or credentials.
   - For long or messy content, separate "must keep," "optional," and "cut" ideas before drafting.

3. Choose the carousel angle:
   - Use a single controlling angle such as how-to, mistake list, myth-versus-truth, before-after, teardown, checklist, framework, case study, lesson recap, or contrarian reframe.
   - Read `references/carousel-patterns.md` when the source is dense, the user asks for multiple angle options, or the draft needs stronger hooks, slide roles, or QA.

4. Build the slide sequence:
   - Slide 1: hook with a specific promise, tension, or curiosity gap.
   - Slide 2: context, stakes, or why the reader should care now.
   - Middle slides: one idea per slide, ordered as steps, reasons, mistakes, myths, examples, or proof.
   - Next-to-last slide: recap, checklist, or save-worthy synthesis.
   - Final slide: clear CTA that matches the user's goal.

5. Draft design-free slides:
   - Use a table with slide number, slide role, headline, body copy, visual cue, and alt text when useful.
   - Keep each slide skimmable: short headline, tight body, one clear takeaway.
   - Visual cues should be plain-language layout or imagery suggestions, not instructions requiring a specific design tool.

6. Add publishing support:
   - Caption or post copy.
   - CTA variants when useful.
   - Optional hashtag or keyword suggestions only when they fit the platform.
   - Optional speaker notes, repurposing ideas, or designer handoff notes if requested.

7. Run carousel QA before finalizing:
   - The hook makes the right audience want slide 2.
   - Every slide advances the same argument.
   - No slide depends on hidden context from the source.
   - Dense slides are split or simplified.
   - Claims are source-faithful and flagged if verification is needed.
   - The CTA is specific and not overbearing.

## Output Patterns

For a full carousel, include:
- Carousel title or angle
- Target platform and assumptions
- Slide-by-slide storyboard table
- Caption or post copy
- CTA options when useful
- Quality notes or claims-to-verify

For an angle pass, include:
- 3 to 5 carousel angles
- Best-fit audience and promise for each angle
- Suggested slide count
- Why each angle works

For an audit, include:
- What works
- Where attention drops
- Slide-level fixes
- Stronger hook or CTA options
- Final revised storyboard if the user asks for one

## Writing Rules

- Write for swiping behavior: curiosity first, clarity always.
- Prefer concrete nouns, active verbs, and reader-facing benefits.
- Avoid generic hooks such as "In today's world" or "Unlock the power of."
- Avoid fake urgency, fake expertise, guaranteed outcomes, and unsupported numbers.
- Keep jargon only when the audience expects it, and define it when clarity would improve.
- Turn paragraphs into decisions, steps, contrasts, examples, or checklists.
- Match the user's voice when source material provides one; otherwise use a concise editorial tone.

## Tool And Format Rules

- Do not open or require design tools unless the user explicitly asks for finished visual production.
- Do not generate images by default. Provide visual direction that a human or later design workflow can use.
- If the user provides a file, inspect the file before drafting.
- If the user provides only a URL and its content is unavailable, retrieve the source when tools allow; otherwise ask for the relevant text.
- If the source includes legal, medical, financial, technical, or compliance claims, preserve uncertainty and mark claims that need expert or source verification.
